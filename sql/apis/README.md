# SQL APIs Documentation

This directory contains SQL documentation files that serve as developer reference for understanding the database queries generated by the ORM (SQLAlchemy) in the application.

## Overview

These SQL files are **documentation/reference only** - they are not directly executed by the application. The actual query logic is implemented in Python using SQLAlchemy ORM in the `app/repositories/` directory. These files illustrate what SQL queries the ORM generates for each API endpoint.

## Naming Conventions

### File Naming

- Use lowercase with underscores: `list_contacts.sql`, `create_company.sql`
- Match the endpoint operation: `list_`, `count_`, `retrieve_`, `create_`, `update_`, `delete_`
- For attribute lists: `list_{attribute_name}.sql` (e.g., `list_titles.sql`, `list_industries.sql`)

### Directory Organization

- Group by API version: `v1/`, `v2/`
- Group by resource: `contacts/`, `companies/`, `users/`, etc.
- Separate core operations from attributes: `core/`, `attributes/`
- Company-specific contact operations: `companies/contacts/`

## File Format Standard

All SQL files follow a standard template (see `TEMPLATE.sql`):

1. **Header Section**: Endpoint path, API version, description
2. **Parameters Section**: Query, path, and body parameters with types and descriptions
3. **Response Structure**: Expected response schema
4. **Response Codes**: HTTP status codes and their meanings
5. **Authentication**: Auth requirements
6. **ORM Query Optimization Notes**: Important notes about conditional JOINs and query optimization
7. **Example Usage**: Example API calls
8. **Query Examples**: SQL queries showing different scenarios

## Key Concepts

### Conditional JOINs

The ORM implementations use **conditional JOINs** to optimize query performance:

- **Minimal queries**: Only join tables when filters require them
- **Company joins**: LEFT JOIN companies only when company filters are present
- **Metadata joins**: LEFT JOIN metadata tables only when metadata filters are present
- **Default ordering**: Uses indexed fields (e.g., `created_at DESC`) that don't require joins

Example from `list_contacts.sql`:

- No filters → Only `contacts` table (no joins)
- Company filter → LEFT JOIN `companies`
- Metadata filter → LEFT JOIN `contacts_metadata` and/or `companies_metadata`

### Query Patterns

1. **List Operations**: `SELECT ... FROM ... WHERE ... ORDER BY ... LIMIT ... OFFSET ...`
2. **Count Operations**: `SELECT COUNT(...) FROM ... WHERE ...`
3. **Retrieve Operations**: `SELECT ... FROM ... WHERE uuid = $1`
4. **Create Operations**: `INSERT INTO ... VALUES (...) RETURNING ...`
5. **Update Operations**: `UPDATE ... SET ... WHERE uuid = $1 RETURNING ...`
6. **Delete Operations**: `DELETE FROM ... WHERE uuid = $1`

### Filter Application Order

The ORM applies filters in a specific order:

1. Contact/Company field filters
2. Company filters (if company join exists)
3. Metadata filters (if metadata joins exist)
4. Special filters (domain, keywords with field control)
5. Search terms (multi-column search)

**Contacts:**
